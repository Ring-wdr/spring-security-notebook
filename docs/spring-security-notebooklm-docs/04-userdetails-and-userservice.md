# Spring Security: UserDetails 및 UserDetailsService 확장 학습 가이드

본 문서는 'SpringSecurity 04 - UserDetail과 UserService 확장' 소스 내용을 바탕으로 작성되었습니다.

---

## 1. 문서 개요
이 문서는 Spring Security의 핵심 인터페이스인 `UserDetails`와 `UserDetailsService`를 사용자 정의(Custom)하여 실제 데이터베이스의 사용자 정보와 연동하는 방법을 다룹니다. 사용자의 데이터를 엔티티(Entity)에서 DTO로 변환하고, 권한(Role)을 스프링 시큐리티 규격에 맞게 매핑하며, 서비스 계층에서 사용자 정보를 로드하는 전체 과정을 상세히 설명합니다 [1], [2].

## 2. 핵심 개념 정리
*   **UserDetails:** 스프링 시큐리티가 사용자의 정보를 인식할 수 있도록 하는 **밸류 오브젝트(VO)**입니다. 본 소스에서는 이를 상속받은 `User` 클래스를 확장하여 `SubscribedDTO`를 구현합니다 [1], [3].
*   **UserDetailsService:** 유저네임(이메일 등)을 기반으로 사용자의 정보를 가져오는 **인증 서비스 인터페이스**입니다. 스프링 시큐리티가 인증 과정에서 이 서비스를 자동으로 호출합니다 [2].
*   **SimpleGrantedAuthority:** 스프링 시큐리티에서 권한을 나타낼 때 사용하는 클래스입니다. 문자열 형태의 권한을 객체화하여 사용합니다 [4].
*   **Claims (클레임):** JWT의 페이로드(Payload)에 담길 데이터 묶음입니다. 본 소스에서는 이메일, 닉네임, 권한 등을 맵(Map) 형태로 정의합니다 [4], [2].
*   **BCrypt:** 패스워드를 안전하게 해싱하기 위한 알고리즘으로, 본 소스에서는 `SHA-256` 기반의 해싱 값을 생성하여 테스트에 활용합니다 [5].

## 3. 요청 흐름 또는 동작 원리
1.  **로그인 요청:** 클라이언트가 폼이나 API를 통해 유저네임과 패스워드를 전달합니다 [6].
2.  **서비스 호출:** 스프링 시큐리티의 인증 프로세스가 시작되면, 등록된 `UserDetailsService`의 `loadUserByUsername` 메서드가 호출됩니다 [2], [7].
3.  **데이터 조회:** 서비스 내에서 리포지토리(`findByEmail`)를 통해 DB에서 엔티티 정보를 읽어옵니다. 이때 `@EntityGraph`를 통해 권한 정보도 함께 가져옵니다 [8], [9].
4.  **객체 변환:** 조회된 엔티티 정보를 `UserDetails` 타입인 DTO 객체로 변환하여 리턴합니다 [10].
5.  **검증:** 리턴된 DTO 내의 패스워드와 클라이언트가 입력한 패스워드를 스프링 시큐리티가 내부적으로 비교하여 인증 여부를 결정합니다. (패스워드 체크 로직은 개발자가 직접 작성하지 않고 시큐리티가 알아서 처리합니다) [2].

## 4. 코드/설정 포인트
*   **DTO 정의:** `SubscribedDTO`는 `org.springframework.security.core.userdetails.User` 클래스를 상속받습니다. 이는 `UserDetails`를 직접 구현하는 것보다 편리하기 때문입니다 [1], [3].
*   **권한 매핑:** 엔티티의 Enum 권한 리스트를 `SimpleGrantedAuthority` 객체 리스트로 변환하여 `super` 생성자에 전달해야 합니다 [4].
*   **@Service 어노테이션:** `UserDetailsService`를 구현한 클래스에 `@Service`를 붙여 빈(Bean)으로 등록해야 스프링 시큐리티가 이를 인지하고 주입(Injection)받을 수 있습니다 [7].
*   **JWT 클레임 생성:** DTO 내부에 `getClaims()` 메서드를 만들어 이메일, 닉네임, 소셜 여부, 권한 등을 맵 형식으로 반환하도록 준비합니다 [4], [2].
*   **로그인 URL 설정:** `SecurityConfig`에서 `formLogin()` 설정을 통해 실제 포스트(POST) 요청을 받을 URL을 지정할 수 있습니다 [11].

## 5. 실무 관점 주의사항
*   **순환 참조 방지:** 엔티티에서 `toString()`을 사용할 때 권한(Roles) 필드를 제외(`exclude`)해야 합니다. 권한 엔티티 내에 다시 사용자 PK가 포함되어 있을 경우 무한 루프가 발생할 수 있기 때문입니다 [8], [1].
*   **아우터 조인 활용:** 권한 정보를 읽어올 때 `Left Outer Join`을 사용해야 합니다. 권한이 없는 사용자도 조회가 되어야 인증 프로세스를 진행할 수 있기 때문입니다 [8].
*   **패스워드 노출 방지:** `getClaims()`를 통해 클라이언트에 전달할 정보에서 패스워드는 가급적 제외하는 것이 보안상 좋으나, 로직상 필요한 경우에만 신중히 포함합니다 [2].
*   **권한 명명 규칙:** 스프링 시큐리티는 기본적으로 `ROLE_` 접두사를 기대하므로, 이놈(Enum) 정의 시 이를 포함하거나 매핑 시 주의해야 합니다 [10].

## 6. 자주 헷갈리는 포인트
*   **유저네임의 의미:** 스프링 시큐리티에서 `Username`은 반드시 사용자의 '이름'을 의미하는 것이 아니라, 식별자인 아이디나 이메일을 의미합니다 [2].
*   **패스워드 체크 주체:** 개발자는 `loadUserByUsername`에서 DB 데이터를 찾아 '제공'만 할 뿐이며, 실제 입력값과 DB값의 일치 여부는 시큐리티 엔진이 판단합니다 [2].
*   **User 클래스 출처:** `User` 클래스는 우리가 만든 엔티티가 아니라 `org.springframework.security.core.userdetails` 패키지에 있는 시큐리티 전용 클래스입니다 [1].
