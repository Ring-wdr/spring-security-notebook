# JWTAuthenticationFilter 구현 및 검증 학습 가이드

본 문서는 'SpringSecurity 06 - JWTAuthenticationFilter 작성하기' 소스 내용을 바탕으로 작성되었습니다.

원본 강의: [SpringSecurity 06 - JWTAuthenticationFilter 작성하기](https://youtu.be/tg-4_glAFUc?si=FSGhT5uaW9-y5Ef5)

---

## 1. 문서 개요
이 문서는 Spring Security에서 JWT(JSON Web Token)를 사용하여 클라이언트의 요청을 검증하는 **`JWTAuthenticationFilter`**(또는 `JWTCheckFilter`)의 구현 과정을 다룹니다. 필터를 통해 특정 경로에 대한 보안 체크를 제외하는 방법, 헤더에서 토큰을 추출하고 검증하는 로직, 그리고 인증 실패 시의 예외 처리 방법을 상세히 설명합니다 [1-3].

## 2. 핵심 개념 정리
*   **JWT 구조 및 보안:** 헤더(타입, 알고리즘), 페이로드(클레임), 시그니처로 구성됩니다. **패스워드는 암호화되어 있더라도 유추가 가능하므로 절대 토큰에 포함해서는 안 됩니다** [4].
*   **OncePerRequestFilter:** `GenericFilterBean`의 단점(요청 당 여러 번 실행될 수 있음)을 보완하여, **요청당 딱 한 번만 실행**되도록 보장하는 스프링 제공 필터 클래스입니다 [5].
*   **shouldNotFilter:** 특정 요청 URI에 대해 현재 필터의 로직을 수행할지 말지를 결정하는 메서드입니다. 여기서 `true`를 반환하면 필터 체크를 건너뛰고 다음 단계로 진행합니다 [1, 2].
*   **AntPathMatcher:** `**`와 같은 와일드카드를 사용하여 경로 패턴을 매칭하는 도구입니다. 복잡한 경로를 효율적으로 체크하기 위해 사용합니다 [6, 7].
*   **Bearer 토큰:** HTTP `Authorization` 헤더에 담겨오는 토큰 형식으로, 실제 토큰 값 앞에 `Bearer `라는 문자열이 붙습니다 [3, 8].
*   **401 vs 403 상태 코드:** 
    *   **401 (Unauthorized):** 로그인 자체가 안 된 경우(인증 실패) [9].
    *   **403 (Forbidden):** 로그인은 되었으나 해당 리소스에 접근할 권한이 없는 경우(인가 실패) [9].

## 3. 요청 흐름 또는 동작 원리
1.  **요청 수신:** 클라이언트가 API를 요청합니다.
2.  **필터 진입:** `JWTAuthenticationFilter`가 작동합니다.
3.  **경로 체크(`shouldNotFilter`):** 로그인, 회원가입, 정적 리소스(이미지 등), 스웨거(Swagger) 등 보안 체크가 필요 없는 경로인지 확인합니다 [1, 2, 7].
    *   제외 경로라면: 즉시 다음 필터로 넘깁니다.
    *   체크 경로라면: 다음 단계를 진행합니다.
4.  **헤더 추출:** `Authorization` 헤더에서 값을 읽어와 `Bearer ` 접두사를 제거하고 실제 JWT 토큰만 끄집어냅니다 [3, 8].
5.  **토큰 검증:** `JWTUtil`을 통해 토큰의 유효성을 검사하고 페이로드(Claims)를 파싱합니다 [3].
6.  **인증 완료 및 전파:** 검증이 성공하면 사용자 정보를 DTO에 담아 시큐리티 컨텍스트에 설정하고 다음 필터로 전달합니다 [9, 10].
7.  **예외 처리:** 토큰이 없거나 유효하지 않으면 `401` 에러 메시지를 JSON 형태로 클라이언트에 반환하고 필터 동작을 종료합니다 [3].

## 4. 코드/설정 포인트
*   **필터 상속:** `extends OncePerRequestFilter`를 사용하여 구현합니다 [5].
*   **필터 순서 설정:** `SecurityConfig`에서 `http.addFilterBefore(new JWTAuthenticationFilter(), UsernamePasswordAuthenticationFilter.class)`를 통해 기존 로그인 필터 앞에 배치합니다 [1, 11].
*   **토큰 파싱:** `header.substring(7)`을 사용하여 앞의 'Bearer ' 7글자를 잘라내고 토큰 본문만 획득합니다 [3, 8].
*   **경로 제외 패턴:** `AntPathMatcher`를 사용하여 `API/subscriber/login`, `/favicon.ico` 등은 필터를 타지 않도록 설정합니다 [6, 7].
*   **메서드 보안 활성화:** `@EnableMethodSecurity`를 설정 클래스에 추가하면 컨트롤러 메서드 레벨에서 `@PreAuthorize("hasAnyRole('...')")` 등을 사용할 수 있습니다 [9, 10].
*   **JSON 응답:** 에러 발생 시 `ObjectMapper`를 사용하여 `Map.of("error", "ERROR_ACCESS_TOKEN")`과 같은 형식으로 클라이언트에게 직접 응답을 보냅니다 [3].

## 5. 실무 관점 주의사항
*   **패스워드 포함 금지:** 암호화된 패스워드라도 토큰에 포함되면 해킹 툴을 통해 원본이 유추될 수 있으므로 절대 넣지 말아야 합니다 [4].
*   **경로 관리의 번거로움:** `shouldNotFilter`에 모든 예외 경로를 일일이 적기 힘들기 때문에 `**` 패턴(AntPathMatcher)을 적극 활용해야 합니다 [6-8].
*   **프론트엔드 대응:** REST 방식에서는 브라우저가 자동으로 토큰을 보내지 않으므로, 엑시오스(Axios) 같은 라이브러리를 통해 요청 헤더에 토큰을 직접 심어주어야 합니다 [10, 12].
*   **상수 사용:** `Authorization` 헤더 이름 등을 하드코딩하기보다 스프링이 제공하는 상수를 사용하는 것이 코드 리뷰 시 권장됩니다 [3].

## 6. 자주 헷갈리는 포인트
*   **GenericFilterBean vs OncePerRequestFilter:** 과거에는 전자를 썼으나, 메모이제이션이나 중복 실행 문제로 인해 현재는 **OncePerRequestFilter** 사용이 표준입니다 [5].
*   **shouldNotFilter 리턴값:** `true`를 반환하는 것이 '필터링을 하지 않겠다(통과시키겠다)'는 의미임을 주의해야 합니다 [1, 5].
*   **토큰 유출 시 대응:** Access Token이 유출되어도 유효기간(예: 10분) 동안은 막을 방법이 없으므로, 유효기간을 짧게 가져가고 Refresh Token 전략을 병행해야 합니다 [1, 4].

## 8. 확인 문제 5개
1. **필터 구현 시 요청당 한 번만 실행되도록 보장하기 위해 상속받는 클래스는 무엇입니까?**
   - (정답: `OncePerRequestFilter`) [5]
2. **JWT 토큰 본문을 추출하기 위해 `Authorization` 헤더 값의 앞에서부터 잘라내야 하는 문자열의 길이는?**
   - (정답: 7자 / 'Bearer '의 길이) [3, 8]
3. **특정 URI 요청 시 필터 검증을 건너뛰도록 설정하는 메서드의 이름은 무엇입니까?**
   - (정답: `shouldNotFilter`) [1, 5]
4. **사용자가 로그인을 하지 않아 누구인지 알 수 없을 때 발생하는 HTTP 상태 코드는 무엇입니까?**
   - (정답: 401 / Unauthorized) [9]
5. **토큰이 유출되었을 때 피해를 최소화하기 위해 소스에서 권장하는 보안 조치는 무엇입니까?**
   - (정답: 패스워드 포함 금지 및 토큰 유효기간을 짧게 설정) [1, 4]
