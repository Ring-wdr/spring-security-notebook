# Spring Security 중간 정리 및 Bearer 토큰 테스트 학습 가이드

본 문서는 'SpringSecurity 07 - 스프링 시큐리티 중간 정리 및 Bearer 토큰을 이용한 테스트 작성' 소스 내용을 바탕으로 작성되었습니다.

---

## 1. 문서 개요
이 문서는 Spring Security의 전체적인 아키텍처를 복습하고, 특히 **JWT 토큰 생성 시 패스워드를 제외하는 보안 처리**와 **인텔리제이(IntelliJ)를 활용한 Bearer 토큰 테스트 방법**을 다룹니다. 실무에서 성능 최적화를 위해 비밀키(Secret Key)를 관리하는 방법과 인증 필터의 경로 예외 설정에 대해서도 상세히 설명합니다 [1-3].

## 2. 핵심 개념 정리
*   **AuthenticationFilter (인증 필터):** 클라이언트의 요청을 가장 먼저 받는 곳으로, `DispatcherServlet`에 도달하기 전 단계에서 동작합니다 [1].
*   **AuthenticationManager:** 전체 인증 과정을 관리하는 '매니저' 역할을 하며, `SecurityConfig`가 이 역할을 수행하는 기준점이 됩니다 [1].
*   **UserDetailsService:** DB에서 사용자 정보를 읽어오는 서비스로, `loadUserByUsername` 메서드를 통해 스프링 시큐리티에 데이터를 전달합니다 [4, 5].
*   **UserDetails:** 시큐리티가 인식하는 사용자 정보 객체입니다. 본 소스에서는 DTO가 `User` 클래스를 상속(extends)받아 이를 구현합니다 [4].
*   **Bearer Token:** HTTP 헤더에 토큰을 담아 보낼 때 사용하는 약속된 형식으로, 토큰 값 앞에 `Bearer `(공백 포함)를 붙입니다 [6].
*   **Signature (시그니처):** 서버에서 토큰의 조작 여부를 확인하는 체크섬으로, 서버의 비밀키가 있어야만 검증이 가능합니다 [2].

## 3. 요청 흐름 또는 동작 원리
1.  **요청 수신:** 클라이언트가 유저네임(이메일)과 패스워드를 담아 요청을 보냅니다 [1].
2.  **필터 처리:** `JWTAuthenticationFilter`가 요청을 가로채서 `Authorization` 헤더의 토큰 유무를 확인합니다 [1, 7].
3.  **데이터 조회:** 인증이 필요한 경우 `UserDetailsService`가 DB에서 사용자 데이터를 셀렉트하여 리턴합니다 [4].
4.  **인증 및 비교:** 스프링 시큐리티가 DB의 패스워드와 입력된 패스워드를 비교합니다. 개발자는 패스워드를 시큐리티에게 '알려주기'만 하면 비교는 시큐리티가 직접 수행합니다 [4, 8].
5.  **핸들러 실행:** 인증 결과에 따라 `SuccessHandler`(성공 시 JWT 발급) 또는 `FailureHandler`(실패 시 JSON 에러 응답)가 호출됩니다 [2, 5, 9].
6.  **토큰 발급:** 성공 핸들러에서 엑세스 토큰(Access Token)을 생성할 때, 보안을 위해 패스워드를 강제로 비운(빈 문자열) DTO를 새로 만들어 사용합니다 [8, 10].

## 4. 코드/설정 포인트
*   **SecurityConfig:** CORS, CSRF 설정 및 각종 핸들러(성공, 실패, 권한 없음)를 등록하는 중앙 관제소입니다 [1, 2, 9].
*   **DTO의 `super` 호출:** DTO 생성자에서 `super(username, password, authorities)`를 호출하여 스프링 시큐리티에게 패스워드 정보를 반드시 전달해야 합니다 [4, 8, 9].
*   **JWTUtil의 Secret Key:** 키는 최소 **256비트(32글자) 이상**이어야 하며, 성능을 위해 매번 해싱하지 않고 `static final` 상수로 관리하는 것이 좋습니다 [3].
*   **Bearer 접두사:** 클라이언트로부터 전달받은 헤더 값에서 `Bearer ` 이후의 토큰 본문만 추출하기 위해 `substring(7)`을 사용합니다 [6, 11].
*   **shouldNotFilter:** 로그인, 가입, 스웨거(Swagger), 정적 리소스 등 토큰 체크가 필요 없는 경로는 필터에서 제외하도록 설정합니다 [7].

## 5. 실무 관점 주의사항
*   **토큰 내 패스워드 제외:** 엑세스 토큰 페이로드에 암호화된 패스워드가 포함되면 해킹의 위험이 크므로, **반드시 제외**해야 합니다 [1, 8].
*   **CPU 사용량 최적화:** 비밀키 생성 시 해싱 작업을 반복하면 CPU 부하가 커지므로, 이를 고정된 상수로 정의하여 재사용해야 합니다 [3].
*   **비밀키 보안:** 솔트(Salt) 역할을 하는 키 정보는 절대 외부로 유출되어서는 안 됩니다 [3].
*   **에러 핸들링:** 토큰이 없거나 조작된 경우 서버 내부 에러가 아닌, 명확한 에러 메시지(예: `ERROR_ACCESS_TOKEN`)를 JSON 형태로 반환해야 클라이언트가 대응할 수 있습니다 [7, 11].

## 6. 자주 헷갈리는 포인트
*   **패스워드를 알려주는 시점 vs 제외하는 시점:** 시큐리티가 인증을 수행할 때는 패스워드를 **알려줘야** 하지만, 인증 후 토큰을 구울(Generate) 때는 패스워드를 **제외**해야 합니다. 이 과정에서 DTO를 새로 생성(`new`)하는 테크닉이 필요합니다 [8, 10].
*   **401 에러 원인:** `shouldNotFilter`에 경로를 등록하지 않으면, 로그인이 필요 없는 페이지임에도 토큰이 없다는 이유로 필터에서 튕겨 나가게 됩니다 [7, 11].
*   **JWT의 가시성:** JWT는 누구나 내용을 열어볼 수 있습니다(`jw.io` 등). 보안의 핵심은 '내용 숨기기'가 아니라 '조작 방지(시그니처)'에 있습니다 [2].

    *   (정답: 스프링 시큐리티 엔진 / 개발자는 정보를 제공만 함) [4, 8]
2.  **JWT의 비밀키(Secret Key)는 최소 몇 비트(또는 몇 글자) 이상이어야 'Weak Key' 에러가 발생하지 않습니까?**
    *   (정답: 256비트 / 32글자 이상) [3]
3.  **인증 필터(`OncePerRequestFilter`)에서 특정 경로를 검증 대상에서 제외할 때 사용하는 메서드 명칭은?**
    *   (정답: `shouldNotFilter`) [7]
4.  **클라이언트가 토큰을 보낼 때 사용하는 'Bearer' 접두사와 토큰 값 사이에는 어떤 문자가 포함되어야 합니까?**
    *   (정답: 한 칸의 공백 / Space) [6]
5.  **토큰 생성 시 패스워드를 제외하기 위해 본 소스에서 사용한 방법은 무엇입니까?**
    *   (정답: 패스워드를 빈 문자열로 설정한 새로운 DTO 객체를 생성하여 전달함) [8, 10]
