# Spring Security 및 JWT 전체 아키텍처 정리 학습 문서

본 문서는 'SpringSecurity 10 - 전체 정리 및 마무리' 소스의 내용을 바탕으로 작성된 최종 요약 가이드입니다.

---

## 1. 문서 개요
이 문서는 Spring Security와 JWT를 연동한 전체 시스템의 구조와 클래스 간의 관계를 정리합니다 [1], [2]. 필터 계층부터 서비스, 핸들러, 유틸리티 클래스에 이르기까지 전체적인 요청 처리 흐름을 복습하고, 특히 리프레시 토큰(Refresh Token)의 갱신 로직과 실무적인 아키텍처 설계 관점을 중점적으로 다룹니다 [3], [4].

## 2. 핵심 개념 정리
*   **JWTAuthenticationFilter:** `OncePerRequestFilter`를 상속받아 구현하며, 요청당 한 번의 인증을 보장하는 필터입니다 [1].
*   **UserDetailsService:** 사용자의 정보를 가져오는 핵심 인터페이스로, 본 소스에서는 `CustomUserDetailService`가 이를 구현합니다 [5].
*   **UserDetails:** Spring Security가 인식하는 사용자 정보 규격이며, DTO 클래스가 이를 확장(extends)하여 사용합니다 [5].
*   **SecurityContextHolder:** 인증된 사용자 정보(`Authentication`)를 보관하는 곳으로, 이를 통해 `@PreAuthorize`와 같은 메서드 보안을 수행할 수 있습니다 [2].
*   **JWTUtil:** 토큰 생성(`generateToken`)과 검증(`validateToken`)을 담당하는 유틸리티 클래스입니다 [5].
*   **유닉스 타임(Unix Time):** `System.currentTimeMillis()`를 1,000으로 나눈 초 단위의 시간으로, 토큰 만료 계산에 사용됩니다 [3].

## 3. 요청 흐름 또는 동작 원리
1.  **필터 진입:** 클라이언트의 요청을 `JWTAuthenticationFilter`가 가로챕니다 [1].
2.  **토큰 검증:** `JWTUtil`을 통해 토큰의 유효성을 확인하고 클레임(Claims)을 추출합니다 [5].
3.  **컨텍스트 등록:** 검증된 정보를 `SecurityContextHolder`에 저장하여 시스템 전역에서 인증 상태를 유지합니다 [2].
4.  **리프레시 토큰 로직:** 
    *   현재 시간을 초 단위로 계산합니다(`System.currentTimeMillis() / 1000`) [3].
    *   리프레시 토큰의 남은 시간(`exp` - 현재 시간)을 계산합니다 [3].
    *   남은 시간이 **60분(3600초)**보다 작으면 새로운 리프레시 토큰을 생성하고, 그렇지 않으면 기존 토큰을 유지합니다 [3].
5.  **응답:** 새로운 엑세스 토큰(Access Token)은 무조건 생성하여 전달하며, 조건에 따라 리프레시 토큰도 함께 반환합니다 [3], [6].

## 4. 코드/설정 포인트
*   **시간 타입 처리:** 연산의 정확성을 위해 `long` 타입을 사용하며, 이는 `System.currentTimeMillis()`의 반환 타입과 일치시키기 위함입니다 [3].
*   **핸들러 등록:** `SecurityConfig`(`CustomSecurityConfig`) 내의 `filterChain` 메서드에서 로그인 성공/실패 핸들러 및 접근 거부(Access Denied) 핸들러를 등록합니다 [5], [4].
*   **테스트 설정:** 토큰 갱신 테스트를 원활하게 하기 위해 유효 기간을 일시적으로 줄여서(예: 10분 → 1분) 확인하는 기법을 사용합니다 [6].
*   **전역 예외 처리:** `CustomControllerAdvice`를 통해 시큐리티 과정에서 발생하는 익셉션을 JSON 형태로 응답합니다 [4].

## 5. 실무 관점 주의사항
*   **구조 파악의 중요성:** 단순히 코드를 외우는 것보다 클래스와 인터페이스 간의 전체적인 그림(아키텍처)을 그릴 수 있는 능력이 중요합니다 [1], [2].
*   **연산 타입 일치:** 서로 다른 데이터 타입 간의 연산은 오류를 유발할 수 있으므로 시간 연산 시 타입을 주의 깊게 관리해야 합니다 [3].
*   **토큰 갱신 시점:** 리프레시 토큰을 무조건 재발급하기보다 만료가 임박한 시점에만 재발급하는 것이 효율적입니다 [3].

## 6. 자주 헷갈리는 포인트
*   **인터페이스와 구현체:** `UserDetailsService`는 인터페이스이며, 실제 로직은 우리가 만든 `CustomUserDetailService`에 들어있습니다 [5].
*   **초 단위 vs 밀리초 단위:** `System.currentTimeMillis()`는 밀리초 단위이므로 JWT의 만료 시간과 비교할 때는 1,000으로 나눠 초 단위로 맞춰야 합니다 [3].
*   **필터 상속 클래스:** 시큐리티 필터는 다양한 종류가 있지만, 본 시스템에서는 `OncePerRequestFilter`를 표준으로 사용합니다 [1].

    *   (정답: 60분 / 1시간) [3]

    *   (정답: `SecurityContextHolder` / `SecurityContext`) [2]
4.  **로그인 성공 시 실행되는 핸들러의 인터페이스 명칭은 무엇입니까?**
